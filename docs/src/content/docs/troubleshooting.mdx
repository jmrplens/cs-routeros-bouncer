## Connection issues

### Cannot connect to MikroTik

:::danger[Error: dial tcp: connection refused]
The bouncer cannot reach the MikroTik API port.
:::

<Steps>

1. **Verify the API service is enabled**

   ```routeros
   /ip/service/print where name=api
   ```

2. **Check the port matches your configuration** (default: 8728, TLS: 8729)

3. **Verify firewall rules allow access** from the bouncer's IP

4. **If using Docker**, ensure the container can reach the router network

</Steps>

### Authentication failed

:::danger[Error: cannot log in]
Wrong username or password.
:::

<Steps>

1. **Verify credentials** in your configuration file

2. **Check the user exists and has the correct group**

   ```routeros
   /user/print where name=crowdsec
   ```

3. **Verify the user's group** has required policies: `api`, `read`, `write`

4. **Check allowed address** restrictions on the user account

</Steps>

### TLS certificate errors

:::danger[Error: x509: certificate signed by unknown authority]
TLS verification failed.
:::

<Steps>

1. **For self-signed certificates**, set the CA path:

   ```yaml
   crowdsec:
     ca_cert_path: "/path/to/ca.pem"
   ```

2. **Verify the certificate** is valid and not expired

3. **Check the hostname** matches the certificate's CN or SAN

</Steps>

## CrowdSec LAPI issues

### Cannot connect to LAPI

:::danger[Error: connection to CrowdSec LAPI failed]
The bouncer cannot reach the CrowdSec Local API.
:::

<Steps>

1. **Verify the LAPI URL**

   ```bash
   curl http://localhost:8080/v1/decisions
   ```

2. **Check the bouncer API key**

   ```bash
   cscli bouncers list
   ```

3. **If CrowdSec runs in Docker**, ensure network connectivity between containers

</Steps>

### No decisions received

:::caution[Bouncer starts but no IPs are banned]
The bouncer connects successfully but the address list stays empty.
:::

Possible causes:

1. **No active decisions** — check with `cscli decisions list`
2. **Origin filtering too restrictive** — check `crowdsec.origins` config
3. **Scenario filtering excludes all** — check `crowdsec.scenarios_containing` / `crowdsec.scenarios_not_containing`
4. **CrowdSec engine not detecting threats** — check CrowdSec logs

## Firewall rule issues

### Rules not created

:::caution[Address list has entries but no firewall rules]
IPs are added to the address list but no firewall rules exist.
:::

<Steps>

1. **Check filter and raw are enabled**

   ```yaml
   firewall:
     filter:
       enabled: true
     raw:
       enabled: true
   ```

2. **Look for errors in logs**

   ```bash
   journalctl -u cs-routeros-bouncer -f
   ```

3. **Verify the user** has `write` policy for firewall operations

</Steps>

### Rules are not in the expected position

:::note[Firewall rules appear at the bottom]
Rules may be placed after existing rules.
:::

<Steps>

1. **Verify** `rule_placement` in configuration. Use `"top"` for highest priority, `"bottom"` to append, or structured placement (`position`, `before_comment`, `after_comment`) when existing router rules must stay first.

2. **Check for dynamic rules** at position 0. Retry only applies to `top` and `position`; the bouncer tries lower positions when RouterOS refuses a dynamic or built-in target.

3. **Check comment anchors** for `before_comment` and `after_comment`. Matching is case-sensitive; `comment_match: "contains"` is a literal substring match, not a regular expression.

4. **Inspect positions manually**

   ```routeros
   /ip/firewall/filter print
   /ip/firewall/raw print
   ```

5. **Check bouncer logs** with `journalctl -u cs-routeros-bouncer -f | grep -i placement`. Missing comment anchors use `fallback` (`top` by default, or `bottom`); out-of-range numeric positions are appended at the bottom.

</Steps>

## Performance issues

### High CPU during reconciliation

:::caution[Router CPU spikes during bouncer startup]
This is expected during initial reconciliation with large address lists.
:::

**Performance benchmarks:**

| List size               | Time                                          | CPU peak          |
| ----------------------- | --------------------------------------------- | ----------------- |
| ~28,700 IPs (full CAPI) | ~58 s wall-clock; ~35–36 s RouterOS bulk work | ~39% observed     |
| No-drift periodic check | ~3–4 s                                        | Returns near idle |

**To reduce impact:**

1. Filter synced decisions with `origins`:

   ```yaml
   crowdsec:
     origins: ["crowdsec", "cscli"]
   ```

2. Schedule bouncer restarts during low-traffic periods

:::note[CPU impact is temporary]
CPU impact is temporary — only while reconciliation is actively adding or removing entries.
:::

### Sustained high RouterOS CPU after startup

:::caution[Router CPU stays high after reconciliation]
This is not expected from address-list entries simply remaining on the router. The bouncer should be mostly idle between CrowdSec stream updates and periodic reconciliation runs.
:::

Check for repeated RouterOS API churn:

```bash
journalctl -u cs-routeros-bouncer --since "30 min ago" | grep -E "already have such entry|attempting reconnect|RouterOS command failed"
```

Signs of correct behavior (what to look for):

<Steps>

1. Cached duplicate ban decisions are skipped without touching RouterOS
2. RouterOS `already have such entry` device errors do not trigger reconnects
3. Router CPU returns near normal idle/traffic levels after reconciliation finishes

</Steps>

If you still see repeated duplicate errors and reconnect attempts, upgrade to the first release whose [changelog](https://github.com/jmrplens/cs-routeros-bouncer/blob/main/CHANGELOG.md#unreleased) includes the RouterOS device-error handling and cache fast-path fixes. Until that release is cut, build from a commit that contains that changelog entry.

## Docker-specific issues

### Container cannot reach router

:::caution[Connection timeout from Docker container]
Docker networking may prevent the container from reaching the router.
:::

<Steps>

1. **Use host networking** in Docker Compose:

   ```yaml
   services:
     cs-routeros-bouncer:
       network_mode: host
   ```

2. **Or ensure the Docker network** can route to the router's IP

3. **Check DNS resolution** if using hostnames

</Steps>

### Permission denied for config file

:::danger[Error: permission denied reading config]
The container user cannot read the mounted config file.
:::

<Steps>

1. **Check file permissions**: `chmod 644 config.yml`

2. **If using Docker secrets**, ensure the secret is properly mounted

3. **Or use environment variables** instead of a config file

</Steps>
